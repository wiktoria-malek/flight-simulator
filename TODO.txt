Now I was thinking: To clarify whether or not we need a State class, I suggest making a table with:

Columns: State — Interface_whatever — Interface_whatever_RFTrack ( three columns is enough: State itself, an interface to a real machine, and an interface to the simulated one)
Rows: All Methods we need from State and Interfaces. Both those already implemented and those to be implemented (e.g., the screens)
This will show us which methods need a dedicated implementation in the interfaces (pretty much all of them, I would say) and which methods make sense to be in the state (i.e. common to all interfaces since they are inherited).

In the table cell corresponding to a "method" x "column" combination, you could briefly describe the purpose of the method.

I mean, yesterday we kind of concluded that "State" was possibly redundant. But now I have some second thoughts.

Remember that one of the objectives of the 'State' class is also to store the acquired data on disk. Ultimately, it is just thanks to the State class and its standardised structure we store on disk, that we can use our BBA tools with both simulated and real data identically. We should not underestimate this function of the State class... !
